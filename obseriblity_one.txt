Claude Code Local Agent Observability Dashboard - Requirements and Architecture

1. Positioning and Background

Claude Code enables developers to authorize an AI agent in the local terminal to read files, modify code, and execute commands.
However, long-running Claude Code sessions can feel like a black box.

Pain point 1: Token and cost opacity
Developers often do not know how many tokens a complex debugging session consumed, or which large files inflated the context.

Pain point 2: Agent loops
When compilation errors occur, the agent may fall into a loop such as:
run command -> receive error -> change code blindly -> run command again -> receive a similar error.

Pain point 3: Lost context
Plain terminal output is hard to review when trying to understand the agent's reasoning path, file reads, file edits, and diffs.

Claude Log Analyzer is designed as a local-first, low-friction observability dashboard for Claude Code sessions.
It parses local logs and turns them into timelines, diffs, token insights, and agent-flow visualizations.

2. Core Architecture: Pluggable Parser System

The project avoids high-friction network interception approaches such as MITM proxies.
Instead, it uses passive file watching and structured extraction.

2.1 Local Data Flow

Run Claude Code:
The developer uses the Claude CLI normally in the terminal.

Tail Watcher:
A lightweight Node.js service incrementally reads Claude Code JSONL log files.

Claude Code Parser:
The parser converts raw log entries into a standard action schema.

WebSocket Server:
The server streams normalized actions to the frontend.

React UI:
The browser UI renders timelines, diff cards, token usage, session comparison, and Agent Flow playback.

2.2 Standard Action Model

Regardless of how complex the underlying log format is, the parser should normalize events into a small set of core actions.

```typescript
type AgentAction =
  | { type: 'CodeRead'; filePath: string; tokens: number; preview?: string }
  | { type: 'CodeWrite'; filePath: string; diff: string }
  | { type: 'TerminalCommand'; command: string; exitCode: number; output: string }
  | { type: 'AgentThought'; text: string };

interface Span {
  id: string;
  timestamp: number;
  durationMs: number;
  tokenUsage: { input: number; output: number; cost: number };
  action: AgentAction;
}
```

3. Claude Code Specific Requirements

Terminal command tracking:
Capture commands such as `npm test`, `grep`, and `ls`, including stdout, stderr, and exit status.
This helps explain how the agent collected environment information and whether it understood failures.

File inspection tracking:
Capture which files the agent read or listed.
This enables context attribution and helps identify large files that increase token usage.

Code modification tracking:
Convert edits and replacements into Git-style diffs whenever possible.
This makes code changes easier to review than raw replacement instructions.

4. Frontend Experience

4.1 Immersive Timeline

The timeline should render rich cards instead of raw JSON nodes.

Terminal cards:
Render terminal commands in a compact terminal-like card.
If `exitCode !== 0`, highlight the card and stderr content.

Diff cards:
Render file changes as red/green diffs using a diff viewer rather than raw edit instructions.

4.2 Token and Cost Attribution

Token analytics should go beyond total session usage.
The UI should help explain whether tokens were spent on large file reads, repeated generated code, tool retries, or long assistant responses.

4.3 Loop Detection

The analyzer should warn when it detects repeated failure patterns, such as:
modify the same file -> run the same test command -> receive a similar error stack.

When this happens, the UI can insert a warning card into the timeline suggesting manual intervention or a more specific prompt.
